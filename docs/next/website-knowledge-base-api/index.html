Website Knowledge Base — Frontend Integration Guide

Overview​

Websites are scraped and stored as a knowledge source for AI assistants. The pipeline runs asynchronously in a background worker. The frontend triggers actions via REST and tracks progress either via WebSocket push or REST polling.

Endpoints​

All endpoints require an authenticated session. Base path: POST /api/v1/websites

Add a website​

POST /api/v1/websites?aiAssistantId=<id>

aiAssistantId is optional. Omit it (or use isDefaultKnowledge: true) to add the website to the global knowledge pool shared across all assistants.

Body

{
  "baseUrl": "https://example.com",
  "displayName": "My Site",
  "crawlType": "ENTIRE_SITE",
  "isDefaultKnowledge": false
}
FieldTypeRequiredNotes
baseUrlstringyesNormalised automatically (trailing slash added, fragment stripped)
displayNamestringnoDefaults to domain name
crawlTypeenumyesSee crawl types below
isDefaultKnowledgebooleannoDefault false

Crawl types

ValueBehaviour
ENTIRE_SITESitemap-first crawl. Fast path (most sites): sitemap parsed synchronously — returns COMPLETED with pages[] populated immediately, all pages active and enqueued for content extraction. Fallback (no sitemap): job enqueued to background worker, returns PENDING. Worker uses HTML crawl for regular sites or Playwright for hash-routing SPAs.
SPECIFIC_PAGESSitemap-first discovery. Returns COMPLETED immediately with all discovered pages inactive — user selects which to activate. Fallback (no sitemap, regular site): homepage link extraction synchronously, same inactive pattern. Fallback (no sitemap, hash-routing SPA): job enqueued to background worker, returns PENDING with requiresAsyncDiscovery: true — worker uses Playwright to discover hash routes, all pages remain inactive. Pages are never processed until the user activates them.
JUST_THIS_PAGESingle page only. Returns COMPLETED immediately with the base URL page enqueued for content extraction.

Response (201)

For sites with a sitemap (fast path), discovery completes synchronously and the response includes pages:

{
  "id": 18,
  "baseUrl": "https://example.com/",
  "displayName": "Example",
  "crawlType": "ENTIRE_SITE",
  "discoveryStatus": "COMPLETED",
  "isDefaultKnowledge": false,
  "pagesDiscovered": 42,
  "pages": [...]
}

For sites without a sitemap (fallback path), the response is minimal and discovery runs in the background:

{
  "id": 18,
  "baseUrl": "https://example.com/",
  "displayName": "Example",
  "crawlType": "ENTIRE_SITE",
  "discoveryStatus": "PENDING",
  "isDefaultKnowledge": false
}

Check discoveryStatus in the response before setting up polling. If it is already COMPLETED, pages are available immediately — no polling needed for discovery. If PENDING, track progress via WebSocket (website.progress event) or poll GET /api/v1/websites/:id.

List websites​

GET /api/v1/websites?aiAssistantId=<id>

Returns all websites for the authenticated user. Pass aiAssistantId to filter to a specific assistant's assigned websites.

Does not include pages. Use this endpoint for rendering the website list/cards. Each website object includes _count.pages (total page count) but not the page rows themselves — call GET /api/v1/websites/:id to get pages for a specific site.

Get website details (with pages)​

GET /api/v1/websites/:id

Returns the website record plus all discovered pages, and a computed discoveryDurationMs field (milliseconds between discoveryStartedAt and discoveryCompletedAt, or null if discovery has not completed).

Safe to call at any point during discovery — pages appear in the response as soon as they are inserted (which happens before discoveryStatus flips to COMPLETED).

Update website settings​

PATCH /api/v1/websites/:id

Updates displayName, crawlType, etc.

Delete website​

DELETE /api/v1/websites/:id

Removes the website, all pages, all content chunks, and all Pinecone embeddings. Irreversible.

Refresh (re-crawl)​

POST /api/v1/websites/:id/refresh

Triggers a full re-crawl. Enqueues a new discovery job.

Page management​

PATCH /api/v1/websites/pages/:pageId           — toggle isActive on one page
PATCH /api/v1/websites/:id/pages/bulk          — bulk activate/deactivate by pageId array
PATCH /api/v1/websites/:id/pages/deactivate-all
PATCH /api/v1/websites/:websiteId/pages/:pageId/deactivate
PATCH /api/v1/websites/:websiteId/pages/:pageId/reactivate
POST  /api/v1/websites/pages/:pageId/reprocess — retry a FAILED page
GET   /api/v1/websites/pages/:pageId/chunks    — get extracted text chunks for a page

AI assistant assignment​

POST   /api/v1/websites/:websiteId/assign/:aiAssistantId
DELETE /api/v1/websites/:websiteId/unassign/:aiAssistantId
GET    /api/v1/websites/ai-assistant/:aiAssistantId/assigned

Status Fields​

discoveryStatus (on the website)​

ValueMeaning
PENDINGWebsite created, job not yet picked up by worker
DISCOVERINGWorker is crawling the site — pages are being inserted into the DB during this phase
COMPLETEDAll pages discovered and enqueued for content extraction
FAILEDDiscovery errored; check discoveryError field

Note: Pages exist in the DB while the status is still DISCOVERING. Do not wait for COMPLETED before rendering the page list — render as soon as pages are present in the GET /websites/:id response.

processingStatus (on each page)​

ValueMeaning
PENDINGPage discovered, waiting for content extraction worker
PROCESSINGWorker is scraping and embedding this page
COMPLETEDContent extracted, chunks created, embeddings stored in Pinecone
FAILEDExtraction failed; check processingError. Can be retried via POST /pages/:pageId/reprocess

syncStatus (on the website)​

ValueMeaning
PENDINGNot yet synced
SYNCEDAll content up to date
FAILEDLast sync attempt failed

Tracking Progress​

Two options: WebSocket (real-time push) or polling (simpler fallback).

Option A — WebSocket (real-time)​

Website progress events are pushed through the existing /messages Socket.IO namespace. No separate connection needed if the frontend is already connected there.

Auth — cookies work automatically​

The WebSocket handshake is an HTTP upgrade request. Browsers include the session_id cookie automatically as long as withCredentials: true is set on the Socket.IO client. No explicit token passing is required.

import { io } from "socket.io-client";

const socket = io("/messages", {
  withCredentials: true, // browser sends session_id cookie automatically
  transports: ["websocket"],
});

The same session_id cookie that authenticates REST requests also authenticates the WebSocket connection. If the user is logged in, the connection is authenticated transparently.

Event: website.progress​

Listen for this event after connecting. It fires on every status transition during discovery and page processing.

socket.on("website.progress", (payload) => {
  console.log(payload);
  // { websiteId, type, status, pagesDiscovered?, pagesProcessed?, pageId? }
});

Payload shape

FieldTypeWhen present
websiteIdnumberAlways
type"discovery" | "page"Always
status"DISCOVERING" | "COMPLETED" | "FAILED"Always
pagesDiscoverednumbertype: "discovery"
pagesProcessednumbertype: "discovery"
pageIdnumbertype: "page"

Recommended usage​

socket.on("website.progress", (payload) => {
  if (payload.websiteId !== websiteId) return;

  if (payload.type === "discovery") {
    updateDiscoveryStatus(payload.status, payload.pagesDiscovered);

    if (payload.status === "COMPLETED" || payload.status === "FAILED") {
      // Re-fetch GET /websites/:id to get the full page list
      refreshWebsite(websiteId);
    }
  }

  if (payload.type === "page") {
    // Re-fetch or update individual page status in your local state
    refreshPage(payload.pageId);
  }
});

Note: Events signal that something changed — re-fetch from REST to get the full up-to-date object rather than trying to derive state from the event payload alone.

Event sequence for ENTIRE_SITE crawl​

Fast path (site has a sitemap) — no WebSocket events needed for discovery:

POST /websites → 201 (discoveryStatus: COMPLETED, pages[] populated)
  ↓  (one event per page as content extraction completes)
website.progress { type: "page", status: "COMPLETED", pageId: X }
website.progress { type: "page", status: "COMPLETED", pageId: Y }
  ...

Fallback path, regular site (no sitemap) — HTML crawl with progressive insertion:

POST /websites → 201 (discoveryStatus: PENDING)
  ↓
website.progress { type: "discovery", status: "DISCOVERING", pagesDiscovered: 1 }
website.progress { type: "discovery", status: "DISCOVERING", pagesDiscovered: 2 }
  ↓  (one event per page as it is crawled — pages available in GET /websites/:id immediately)
website.progress { type: "discovery", status: "COMPLETED", pagesDiscovered: N }
  ↓  (one event per page as content extraction completes)
website.progress { type: "page", status: "COMPLETED", pageId: X }
  ...

Fallback path, hash-routing SPA (no sitemap) — Playwright crawl, batch insert:

POST /websites → 201 (discoveryStatus: PENDING)
  ↓
website.progress { type: "discovery", status: "DISCOVERING" }
  ↓  (Playwright renders the site — no incremental events during this phase)
website.progress { type: "discovery", status: "COMPLETED", pagesDiscovered: N }
  ↓  (one event per page as content extraction completes — ENTIRE_SITE only)
website.progress { type: "page", status: "COMPLETED", pageId: X }
  ...

Option B — Polling​

Poll GET /api/v1/websites/:id to track discovery and page processing state. All status fields are available in the REST response.

Recommended polling strategy (Option B)​

const postResponse = await fetch('/api/v1/websites', { method: 'POST', body: ... }).then(r => r.json());

// Fast path: sitemap found — discovery already complete, pages available immediately
if (postResponse.discoveryStatus === 'COMPLETED') {
  renderPageList(postResponse.pages);
  // Still poll for page processingStatus updates
} else {
  // Fallback path: no sitemap — poll for discovery progress
}

const poll = setInterval(async () => {
  const website = await fetch(`/api/v1/websites/${postResponse.id}`).then(r => r.json());

  // Update page list as soon as pages appear (before COMPLETED)
  renderPageList(website.pages);

  // Stop when terminal state reached
  if (website.discoveryStatus === "COMPLETED" || website.discoveryStatus === "FAILED") {
    clearInterval(poll);
  }
}, 3000); // every 3 seconds

Suggested UI states by discoveryStatus​

discoveryStatusSuggested UI
PENDINGSpinner, "Queued…"
DISCOVERINGSpinner, "Scanning website…" — render any pages that appear
COMPLETEDShow full page list; each page shows its own processingStatus
FAILEDShow error state on the website card; check discoveryError

Suggested UI states by page processingStatus​

processingStatusSuggested UI
PENDINGGrey badge, "Waiting…"
PROCESSINGSpinner badge
COMPLETEDShow wordCount, chunkCount
FAILEDError badge + "Retry" button (POST /pages/:pageId/reprocess)

Page Object Fields​

Fields worth rendering:

FieldNotes
urlPage URL
titleExtracted <title> — may be empty string if not found
depth0 = base URL, 1 = directly linked, etc.
processingStatusPENDING / PROCESSING / COMPLETED / FAILED
isActiveWhether this page is included in AI responses. Toggle with PATCH endpoints.
wordCountAvailable once processingStatus: COMPLETED
characterCountAvailable once processingStatus: COMPLETED
chunkCountNumber of text chunks indexed in Pinecone
lastCheckedAtWhen the page was last scraped
processingErrorError message if processingStatus: FAILED

Error Responses​

StatusWhen
400Invalid URL, duplicate website, malformed body
404Website or page not found (or belongs to another user)
409Website with this URL already exists

Error shape:

{
  "statusCode": 400,
  "message": "Website already exists",
  "error": "Bad Request"
}
